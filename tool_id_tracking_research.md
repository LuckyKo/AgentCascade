# Tool Call ID Tracking — Complete Flow Analysis & Fix Plan

## Executive Summary

The `tool_call_id` (e.g., `"call_abc123"`) generated by the LLM is **lost** between step 3 (FUNCTION message creation) and step 6 (async result injection). It is available at the tool execution loop but never threaded through `_handle_call_agent()` or into the async result pipeline.

---

## Complete Flow Trace

### Step 1: LLM generates `tool_use` with `tool_call_id`
**Location:** Inside the LLM API response, stored in the assistant message dict/object.

```python
# Assistant message structure (from LLM API response):
{
    "role": "assistant",
    "content": None,
    "function_call": {"name": "call_agent", "arguments": "..."},
    "extra": {
        "function_id": "call_abc123",   # <-- This IS the tool_call_id
        ...
    }
}
```

### Step 2: Engine detects tool and calls `_execute_tool()`
**Location:** `execution_engine.py` line ~1348

```python
tool_result = self._execute_tool(instance, tool_name, tool_args, llm_messages)
```

The `extra_data` is NOT passed to `_execute_tool`. It's extracted later at line 1405.

### Step 3: FUNCTION message created with `function_id`
**Location:** `execution_engine.py` lines ~1405-1427

```python
# Line 1405 — extra_data IS available here (but NOT passed to _handle_call_agent)
extra_data = out.get('extra', {}) if isinstance(out, dict) else (getattr(out, 'extra', None) or {})

# Lines 1416-1427 — FUNCTION message gets the function_id correctly
fn_msg = Message(
    role=FUNCTION,
    name=tool_name,
    content=tool_result,
    extra={
        'function_id': extra_data.get('function_id', '1'),  # <-- Correctly stored here
        'tool_success': _tool_success,
    },
)
```

**Key issue:** `_handle_call_agent()` is called at line 1348 — BEFORE this extraction. The `function_id` is not available inside `_handle_call_agent()`.

### Step 4: Child runs in background thread
**Location:** `_handle_call_agent()` lines ~1906-1922

```python
# Line 1906 — A DIFFERENT call_id is generated (NOT the tool_call_id)
call_id = f"{instance_name}_{time.monotonic()}"

# Line 1916 — register_async_call uses this synthetic call_id
self.pool.register_async_call(caller_name, call_id)

# Line 1920-1921 — submit_parallel receives the synthetic call_id
result = self.pool.submit_parallel(
    agent_class, instance_name, args, messages, instance.instance_name, child_depth, call_id=call_id
)
```

**Critical:** `call_id` is a synthetic ID (`{instance_name}_{time.monotonic()}`), NOT the LLM's `tool_call_id`. There is NO mapping between them.

### Step 5: When child completes — result goes into async buffer
**Location:** `agent_pool.py` lines ~1675-1681 (inside `task_wrapper`)

```python
completion_msg = f"[Parallel Agent '{instance_name}' Finished]:\n{result}"
self.pool.add_async_result(caller, completion_msg)  # Just a plain string!
```

And in `_notify_async_error()` (line ~1530):
```python
error_msg = f"[Parallel Agent '{instance_name}' Failed]: {error_msg}"
self.pool.add_async_result(caller, error_msg)  # Also just a plain string!
```

### Step 6: Parent drains results and injects as USER message
**Location:** Three injection points in `execution_engine.py`:

#### 6a. SLEEPING guard drain (line ~382-392):
```python
for result in async_results:
    result_msg = Message(
        role=USER, 
        content=f"[BACKGROUND TOOL RESULT]: {result}"   # <-- tool_call_id NOT included!
    )
```

#### 6b. Stable-state drain loop (line ~481-487):
```python
for result in more_results:
    result_msg = Message(role=USER, content=f"[BACKGROUND TOOL RESULT]: {result}")
```

#### 6c. Safety drain in `_post_turn_checks` (line ~1620-1626):
```python
for result in final_drain:
    result_msg = Message(role=USER, content=f"[BACKGROUND TOOL RESULT]: {result}")
```

## Data Flow Diagram

```
LLM API Response
├── assistant message
│   └── extra.function_id = "call_abc123"  ← tool_call_id EXISTS here
│
└── _execute_tool() called (line 1348)
    └── _handle_call_agent() called (line 1784)
        │  [tool_call_id NOT passed — only args, messages, instance]
        │
        ├── call_id = f"{instance_name}_{time.monotonic()}"  ← synthetic ID created
        ├── register_async_call(caller, call_id)             ← registers synthetic ID
        └── submit_parallel(..., call_id=call_id)            ← passes synthetic ID

Child Thread (task_wrapper):
├── agent completes
├── completion_msg = f"[Parallel Agent '{name}' Finished]:\n{result}"  ← plain string
├── add_async_result(caller, completion_msg)                            ← no metadata
└── complete_async_call(caller, call_id)                                ← marks synthetic ID done

Parent Thread (drain):
├── drain_async_results(caller) → ["[Parallel Agent 'Researcher' Finished]:\n..."]
└── Message(content=f"[BACKGROUND TOOL RESULT]: {result}")              ← tool_call_id LOST!
```

---

## Root Cause Analysis

The `tool_call_id` is lost because:

1. **`_handle_call_agent()` never receives the function_id.** It's called at line 1348 before the extra_data extraction at line 1405. The function signature is `_execute_tool(instance, tool_name, tool_args, messages)` — no `function_id` parameter.

2. **The synthetic `call_id` is unrelated to `tool_call_id`.** Line 1906 generates `f"{instance_name}_{time.monotonic()}"` which has no connection to the LLM's `"call_abc123"`.

3. **AsyncResultBuffer only stores strings.** `add_async_result(instance_name, result)` takes a plain string — no metadata slot for tool_call_id.

4. **Drain points format results as plain strings.** All three injection sites use `f"[BACKGROUND TOOL RESULT]: {result}"` with no lookup mechanism.

## Fix Plan

### Option A (Recommended): Store mapping in pool, pass through to `_handle_call_agent`

This is the cleanest approach because it keeps the mapping centralized and uses existing infrastructure.

#### Changes needed:

**1. Add tool_call_id → call_id mapping in `agent_pool.py`:**

Add a new dict alongside `_async_pending_calls`:
```python
# In AgentPool.__init__ (after line 298):
self._call_id_to_tool_call_id: Dict[str, str] = {}  # synthetic call_id → LLM tool_call_id
```

**2. Pass `function_id` into `_handle_call_agent()`:**

Modify the call chain to thread the function_id through:

In `execution_engine.py` line ~1348, change from:
```python
tool_result = self._execute_tool(instance, tool_name, tool_args, llm_messages)
```
to (extract extra_data first):
```python
extra_data = out.get('extra', {}) if isinstance(out, dict) else (getattr(out, 'extra', None) or {})
function_id = extra_data.get('function_id')  # This is the LLM's tool_call_id

tool_result = self._execute_tool(instance, tool_name, tool_args, llm_messages, function_id=function_id)
```

Then modify `_execute_tool` signature to accept `function_id=None`, and pass it through:
```python
def _execute_tool(self, instance, tool_name, tool_args, messages, function_id=None):
    ...
    if tool_name == 'call_agent':
        result = self._handle_call_agent(resolved, messages, instance, function_id=function_id)
```

**3. In `_handle_call_agent()`, register the mapping:**

After generating `call_id` (line 1906), add:
```python
if function_id:
    self.pool._call_id_to_tool_call_id[call_id] = function_id
```

And in `task_wrapper()` in `agent_pool.py`, when adding async results, look up the tool_call_id and include it in the message:
```python
# Before line 1675, add lookup:
tool_call_id = None
if call_id:
    tool_call_id = self.pool._call_id_to_tool_call_id.get(call_id)

completion_msg = f"[Parallel Agent '{instance_name}' Finished]"
if tool_call_id:
    completion_msg += f" for tool_call_id={tool_call_id}"
completion_msg += f":\n{result}"

self.pool.add_async_result(caller, completion_msg)
```

**4. Clean up the mapping when call completes:**

In `task_wrapper()` after `complete_async_call()`:
```python
if call_id:
    self.pool._call_id_to_tool_call_id.pop(call_id, None)
```

**5. Also handle `_notify_async_error()` path:**

Same lookup + formatting in the error path (line ~1527-1530).

### Option B (Alternative): Augment AsyncResultBuffer with metadata

Instead of storing just strings, store tuples/dicts:
```python
class AsyncResultEntry:
    def __init__(self, result: str, tool_call_id: Optional[str] = None):
        self.result = result
        self.tool_call_id = tool_call_id
```

This requires more invasive changes to `AsyncResultBuffer`, `add_async_result()`, `drain_async_results()`, and all three injection points. **Not recommended** — option A is less disruptive.

## Edge Case: Multiple call_agent calls in one turn

If the LLM makes two `call_agent` calls in one turn (e.g., launches Researcher and Coder simultaneously):

```
Assistant message:
- function_call: call_agent(instance_name="researcher", ...) extra.function_id="call_abc123"
- function_call: call_agent(instance_name="coder", ...) extra.function_id="call_def456"
```

**With Option A fix:**
- `register_async_call("Parent", "researcher_1.23")` + mapping `"researcher_1.23" → "call_abc123"`
- `register_async_call("Parent", "coder_4.56")` + mapping `"coder_4.56" → "call_def456"`
- When researcher completes: lookup `"researcher_1.23"` → `"call_abc123"` → inject with correct ID
- When coder completes: lookup `"coder_4.56"` → `"call_def456"` → inject with correct ID

**This is handled correctly** because each synthetic `call_id` is unique and the mapping is per-call-id (not per-instance).
